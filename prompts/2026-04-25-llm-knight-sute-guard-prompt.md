# `wolfbot` 2026-04-25 LLM 騎士・捨て護衛判断強化プロンプト

この文書は、このリポジトリの `wolfbot` を今回の確認事項に合わせて更新するための固定プロンプトです。下の `## Prompt` セクションを、そのまま別の LLM / coding agent に渡すことを想定しています。

今回の主眼は、LLM player が「捨て護衛」などの熟練した人狼プレイヤーが使う用語を認識し、特に騎士として夜の護衛先を選ぶときに、鉄板護衛だけでなく捨て護衛・GJ 狙い・次夜の護衛制限まで比較できるようにすることです。

参考として、一般的な人狼用語と戦術説明を確認済みです。捨て護衛は文脈により「噛まれにくい位置をあえて護衛する」「吊り先や低襲撃価値位置を護衛して噛み合わせを暴く」「連続護衛不可の環境で次夜の本命護衛余地を残す」などの意味で使われます。この bot では騎士の夜行動が合法候補から 1 名を選ぶ設計なので、「誰も守らない」ではなく「合法候補の中から噛まれにくい低価値先をあえて護衛する戦術」として扱います。

## Prompt

```md
あなたは、このリポジトリで直接コードを修正する Python エンジニアです。

目的:
- Discord 人狼 bot `wolfbot` の LLM player を、騎士の護衛判断においてより強い熟練者として振る舞わせる。
- LLM player が `捨て護衛`、`鉄板護衛`、`変態護衛`、`護衛読み`、`護衛誘導`、`連ガ無し / 連続護衛不可` などの熟練人狼用語を認識できるようにする。
- LLM player が騎士で守り先を考えるとき、単に真寄り情報役を守るだけでなく、捨て護衛、GJ 狙い、襲撃予測、翌夜に同じ相手を守れない制約を比較できるようにする。
- ただしゲームルール、夜行動の合法候補、連続護衛不可ルール、DB schema、Discord UI は変更しない。

今回必ず対応すること:
1. 共通ルール block に、熟練用語として `捨て護衛`、`鉄板護衛`、`変態護衛`、`護衛読み`、`護衛誘導`、`連ガ無し / 連続護衛不可` の意味を追加すること。
2. `捨て護衛` は、この bot では「合法候補の中から噛まれにくい低価値先をあえて護衛し、次夜の本命護衛余地を残す、または噛み筋を読むための戦術」と定義すること。
3. `捨て護衛` を `intent=skip`、未提出、対象なし、誰も守らない行動として実装しないこと。この bot の騎士は夜に合法候補から 1 名を選ぶ。
4. 騎士 role-specific strategy に、夜の護衛先を `護衛価値 / 襲撃されやすさ / GJ が縄に与える価値 / 次夜の連続護衛不可リスク / CO 時の説明可能性` で比較する指針を追加すること。
5. 騎士の夜行動 task (`KNIGHT_GUARD`) に、候補ごとに鉄板護衛・捨て護衛・GJ 狙いを比較してから選ぶよう明記すること。
6. 人狼や狂人に、実際の騎士位置や護衛先を知っている前提の情報を渡さないこと。
7. 既存の情報秘匿、role-specific strategy 分離、persona 話法 block、structured output 制約を壊さないこと。

最重要ルール:
- まず既存実装を読み、現在の prompt 構築、LLM 夜行動、騎士の合法護衛候補、既存テストを把握してから修正すること。
- `domain/` は純粋ロジックのまま保つこと。今回の変更で夜処理、勝敗判定、状態遷移を変えない。
- LLM prompt 構築は `src/wolfbot/llm/prompt_builder.py` と必要最小限の prompt tests に閉じること。
- `src/wolfbot/prompts/llm_system_prompt.md` の構造を大きく変えず、原則として既存の `{game_rules_block}` と role-specific strategy に含めること。
- user context に新しい盤面集計データ、護衛候補評価表、CO parser、騎士推定表を追加しないこと。
- Discord channel history を直接拾って prompt に入れてはいけない。LLM 文脈は既存どおり DB の public/private logs からだけ構築すること。
- `game_id` / `audience_seat` ベースの情報分離を壊さないこと。
- 実装後は必ずテスト・lint・型チェックを走らせ、結果を報告すること。

参考にした一般的な用語整理:
- `騎士` と `狩人` は、人狼界隈で同じ護衛役を指すことが多い。この bot の正式役職名は `騎士` だが、公開ログ上で `狩人` や `狩` と呼ばれても同義語として理解してよい。
- `鉄板護衛`: 真寄り情報役、確白寄り、進行役など、襲撃価値が高く村にとって失うと痛い位置を堅く守る護衛。
- `変態護衛`: セオリー上の本命から外れた位置を、襲撃読みで守る護衛。成功すれば強いが、外すと重要役職を抜かれるリスクがある。
- `捨て護衛`: 噛まれにくい、または噛まれても村損失が小さい位置をあえて護衛する戦術。連続護衛不可の環境では、今日本命を守ると明日その本命を守れなくなるため、次夜の本命護衛余地を残す目的でも使う。
- `護衛読み`: 人狼がどこを噛みたいか、どこが護衛されていると読んで噛みを避けるかを推理すること。
- `護衛誘導`: 昼発言で騎士の護衛先に影響を与えようとする発言。村利の整理の場合も、人狼の誘導の場合もある。
- `連ガ無し / 連続護衛不可`: 同じ相手を 2 夜連続で護衛できないルール。この bot は連続護衛不可。

## 1. 共通ルール block に護衛関連の熟練用語を追加する

必要な仕様:
- すべての LLM player が、公開ログ上に `捨て護衛`、`鉄板護衛`、`変態護衛`、`護衛読み`、`護衛誘導`、`連ガ無し`、`狩人`、`狩` などが出ても意味を理解できること。
- 用語説明は「辞書」ではなく、公開ログを読むときの判断ルールとして書くこと。
- 用語説明は実際の役職や護衛先を漏らすものではない。あくまで公開発言・投票・襲撃結果を読むための語彙として扱う。
- `捨て護衛` は、この bot の実装に合わせて「合法候補の中から低襲撃確率・低損失位置を選ぶ」と明記すること。
- `捨て護衛` を未提出、対象なし、skip と混同させないこと。

このタスクで固定する実装方針:
- `src/wolfbot/llm/prompt_builder.py` の `_build_game_rules_block()` に文面を追加すること。
- 既存の `GJ / 平和`、`噛み筋`、`縄計算`、`CO 評価` の近くに置くとよい。
- `build_system_prompt()` の公開引数は増やさないこと。
- DB、状態遷移、夜行動解決、候補生成は変えないこと。

共通ルール block に必ず含める文面趣旨:

- `鉄板護衛`: 真寄り情報役、確白寄り、進行役など、襲撃価値が高く失うと痛い位置を堅く守ること。

- `捨て護衛`: 噛まれにくい、または噛まれても村損失が小さい位置をあえて護衛する戦術。この bot では合法候補から 1 名を選ぶ行動であり、未提出や誰も守らない行動ではない。
- `連ガ無し / 連続護衛不可`: 同じ相手を連続で護衛できないこと。この bot では前夜の護衛先は今夜の合法候補から外れる。
- `護衛読み`: 人狼がどこを噛みたいか、どこが護衛されそうで避けるかを推理すること。

## 2. 騎士 strategy を捨て護衛込みの熟練判断に更新する

`_ROLE_STRATEGIES[Role.KNIGHT]` に、以下の趣旨を必ず追加すること。

必要な仕様:
- 騎士 LLM は夜の護衛先を、単に最も白い相手や真寄り情報役に固定し続けるのではなく、盤面ごとの価値で比較する。
- 騎士 LLM は、同じ相手を連続護衛できないルールを戦術判断に入れる。
- 騎士 LLM は、捨て護衛を「弱い行動」として常に拒否せず、熟練者の選択肢として検討する。
- ただし捨て護衛を常用しない。真寄り占い師・真寄り霊媒師・確白寄り進行役が今夜噛まれると村が崩れる場合は、鉄板護衛を優先する。
- 捨て護衛を選ぶ場合は、なぜ本命が今夜噛まれにくいのか、なぜ次夜に本命を守る余地を残したいのか、短く説明できる状態にする。
- 平和が出た朝や CO する局面では、護衛履歴を日付順に出し、捨て護衛を含む場合も理由を説明できるようにする。

騎士 strategy に必ず含める比較軸:
- `護衛価値`: 真寄り占い師、真寄り霊媒師、確白寄り、進行役、強い村目、狼にとって邪魔な位置か。
- `襲撃されやすさ`: 人狼が今夜実際に噛みそうか。護衛が入りそうで避けられそうか。
- `GJ 価値`: 平和が出ると縄が増えるか、PP/RPP を遠ざけるか、CO で盤面を詰められるか。
- `次夜制約`: 今日守った相手は明日守れない。明日のほうが本命襲撃が濃いなら、今日の護衛をずらす価値があるか。
- `説明可能性`: 後で CO したとき、護衛履歴が合法で、盤面に沿った理由を説明できるか。

文面例:

```text
- 夜の護衛先は、候補ごとに「護衛価値」「襲撃されやすさ」「GJ が縄に与える価値」「次夜の連続護衛不可リスク」「CO 時の説明可能性」を比較して選ぶ。
- 鉄板護衛は、真寄り情報役・確白寄り・進行役など、今夜抜かれると村が大きく崩れる位置を堅く守る選択。
- 捨て護衛は、噛まれにくい低損失位置をあえて護衛し、次夜に本命を護衛できる余地を残す選択。この bot では合法候補から 1 名を選ぶ行動で、未提出や対象なしではない。
- 捨て護衛は、今夜本命が噛まれにくい理由があり、明日以降に本命襲撃が濃くなると読める場合だけ検討する。重要役職が今夜抜かれるリスクが高いなら鉄板護衛を優先する。
```

注意:
- 騎士 strategy に人狼専用の `相方`、`襲撃先を揃える` などを入れないこと。
- 騎士自身に、実際の人狼位置や襲撃先を知っている前提の文面を書かないこと。あくまで公開ログと自分の私的護衛履歴からの推理に留める。
- `捨て護衛` を推奨しすぎて、毎夜低価値先を守る LLM にならないよう、鉄板護衛との使い分けを明記すること。

## 3. `task_night_action(KNIGHT_GUARD)` を強化する

`task_night_action(kind, candidate_tokens)` で `kind is SubmissionType.KNIGHT_GUARD` の場合、現在の合法候補提示に加えて、騎士用の護衛判断チェックリストを入れること。

必要な仕様:
- 候補トークンから 1 名を厳密に選ぶルールは維持する。
- `target_name` は候補トークンと完全一致させる既存制約を維持する。
- 騎士にだけ、護衛先選択前に以下を短く比較するよう指示する。
  - 鉄板護衛すべき本命がいるか。
  - 人狼が今夜そこを噛む可能性は高いか、それとも護衛を読んで避けそうか。
  - 捨て護衛で次夜の本命護衛余地を残す価値があるか。
  - GJ が出た場合に縄や CO 判断へ与える価値は大きいか。
  - その護衛履歴を後で合法かつ自然に説明できるか。
- `intent=night_action` と合法 `target_name` を返させる。捨て護衛を理由に `skip` させない。

文面例:

```text
 騎士の護衛では、候補ごとに「鉄板護衛すべき価値」「今夜実際に噛まれそうか」「GJ の価値」「次夜に同じ相手を守れないリスク」「捨て護衛で本命護衛余地を残す価値」を比較してください。捨て護衛を選ぶ場合も、この bot では合法候補から 1 名を選びます。
```

実装方針:
- `task_night_action()` の `kind` ごとの `extra` を分岐させる。
- WOLF_ATTACK 向け文面、SEER_DIVINE 向け文面、既存候補トークン制約は壊さない。
- prompt 文面だけを変更し、`LLMAction` schema は変更しない。

## 4. 情報秘匿と role leak を壊さない

必要な仕様:
- 非人狼 prompt に、人狼相方情報や人狼専用襲撃連携語彙が混ざってはいけない。
- 騎士 prompt に、実際の人狼位置や今夜の襲撃先を知っている前提を入れてはいけない。
- 人狼 prompt に、実際の騎士 seat、前回護衛先、今回護衛先を漏らしてはいけない。
- 共通ルールの護衛用語は、公開ログ上の語彙理解として全 LLM player に見せてよい。
- 騎士の実践的な護衛選択 advice は `_ROLE_STRATEGIES[Role.KNIGHT]` と `KNIGHT_GUARD` task に置く。

やってはいけないこと:
- `legal_guard_targets()` の仕様変更
- 連続護衛不可ルールの変更
- 騎士が `skip` できる新仕様の追加
- 夜 action なしを捨て護衛として扱う変更
- DB schema 追加
- Discord UI / DM view 変更
- CO parser、護衛 parser、自動盤面分類器の追加
- 実役職一覧や実護衛先を user context に追加
- Discord API から message history を直接 prompt に流す
- 無関係な refactor
