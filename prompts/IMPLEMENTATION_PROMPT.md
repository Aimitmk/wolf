# Discord 人狼 Bot 実装プロンプト

## 目的
このリポジトリに、Discord で 9 人村の人狼を遊べる Python 製 bot を実装してください。人間プレイヤーが 9 人未満の場合は、xAI Grok API を使う LLM プレイヤーで不足人数を補完してください。

この文書は仕様を固定するための実装指示書です。勝手に仕様を省略したり、曖昧な部分を別案に置き換えたりせず、この文書に合わせて実装してください。状態遷移、夜処理、権限管理、再起動復旧の正確さを最優先にしてください。

## 固定要件
- 言語は Python 3.11。
- パッケージ管理と仮想環境は `uv` を使う。
- Discord ライブラリは `discord.py` 系を使う。
- 品質担保として `pytest`, `ruff`, `mypy` を必須にする。
- ゲームは 9 人村専用。配役は固定で `人狼2 / 狂人1 / 占い師1 / 霊媒師1 / 騎士1 / 村人3`。
- 人間は VC で会話しながら遊ぶ。LLM がいる場合のみテキスト議論も併用する。
- LLM プレイヤーは Gnosia のキャラ名と対応する性格を使う。
- 死亡後に暇にならないよう、死者専用の天国テキスト部屋を作る。
- 公開チャンネルで死亡者が発言できないようにする。
- ゲーム状態は SQLite に永続化する。
- bot 再起動後に進行中ゲームを復元できるようにする。

## 非目標
- VC の音声認識や自動文字起こしは実装しない。
- Web UI は作らない。
- 9 人村以外の配役は実装しない。
- 大規模な設定画面は作らない。必要最小限の slash command と DM UI に絞る。

## 技術方針

### 環境構築
以下を前提にプロジェクトを作成すること。

```bash
uv venv --python 3.11 .venv
source .venv/bin/activate
```

Python 3.11 がない場合は以下を使ってよい。

```bash
uv python install 3.11
uv venv --python 3.11 .venv
```

依存関係は少なく保つが、少なくとも以下のカテゴリを満たすこと。
- 本番依存: `discord.py`, `aiosqlite` もしくは同等の非同期 SQLite アクセス手段, `pydantic`, `python-dotenv`, xAI API 用クライアント, `httpx`, `tenacity`
- 開発依存: `pytest`, `pytest-asyncio`, `ruff`, `mypy`

### xAI API
- xAI API のモデル指定は環境変数 `XAI_MODEL` で切り替え可能にする。
- 既定値は `grok-4-1-fast` にする。
- `grok-4-1-fast` は reasoning 可能な fast 系モデルとして扱うが、`reasoning_effort` のような未対応パラメータは送らないこと。
- API キーは `XAI_API_KEY` を使う。
- LLM 応答はできるだけ JSON schema / structured output で受け取り、文字列パースに依存しない。

## Discord インターフェース

### 環境変数
最低でも以下を読むこと。

```env
DISCORD_TOKEN=
XAI_API_KEY=
XAI_MODEL=grok-4-1-fast
DISCORD_GUILD_ID=
MAIN_TEXT_CHANNEL_ID=
MAIN_VOICE_CHANNEL_ID=
```

### Slash Command
コマンドグループは `/wolf` に統一すること。最低限以下を実装すること。
- `/wolf create`
  - 新しい 9 人村の部屋を作る
  - メインテキスト/VC の紐付けを初期化する
  - 死者用チャンネルと人狼専用チャンネルを準備する
- `/wolf join`
  - 人間プレイヤーが参加する
- `/wolf leave`
  - ゲーム開始前のみ退出可
- `/wolf start`
  - 人数不足なら LLM を追加して開始する
- `/wolf status`
  - 現在フェイズ、残り時間、参加者、生死、未提出者数を表示する
- `/wolf extend`
  - `WAITING_HOST_DECISION` 中のみ、現在フェイズの締切を延長する
- `/wolf force-skip`
  - `WAITING_HOST_DECISION` 中のみ、未提出者を棄権または行動なしとして確定する
- `/wolf abort`
  - 現在のゲームを強制終了する

### 秘密操作
- 秘密投票と夜行動は bot DM で行う。
- DM では select menu または明確なテキストコマンド UI を使い、対象選択を誤りにくくする。
- 秘密情報を guild 内公開チャンネルへ送らないこと。
- DM が使えないユーザーにはエラーを返し、ゲーム開始前に検知できるようにする。

### チャンネル運用
- メイン text とメイン VC は既存チャンネルを利用する。
- 天国部屋は bot が専用 text チャンネルを作る。
- 人狼専用チャットは bot が専用 text チャンネルを作る。
- 昼は生存者だけがメイン text に発言可能。
- 死亡者はメイン text を read-only にし、天国部屋にのみ発言可能にする。
- 死亡した人狼はその時点で人狼専用チャットへの書き込み権限を失う。

## ゲーム仕様

### 配役
- 人狼: 2
- 狂人: 1
- 占い師: 1
- 霊媒師: 1
- 騎士: 1
- 村人: 3

### 勝利条件
- 村人陣営勝利: 生存人狼数が 0
- 人狼陣営勝利: 生存人狼数が生存非人狼人数以上
- 勝利判定は以下の 2 箇所で必ず行うこと
  - 昼の処刑解決直後
  - 夜の襲撃解決直後

### LLM 補完
- 人間プレイヤーが 9 人に満たない場合のみ、足りない人数分の LLM プレイヤーを追加する。
- LLM プレイヤーには以下の条件を課すこと。
  - ゲーム中の表示名は Gnosia キャラ名

  - 役職とは独立にランダム割当
  - 発言はメイン text または天国 text のみ。VC には参加しない
- LLM の議論入力はメイン text の投稿内容だけを使う。音声内容は使わない

## フェイズと状態機械

### 状態一覧
状態は最低でも以下を持つこと。
- `LOBBY`
- `SETUP`
- `NIGHT_0`
- `DAY_DISCUSSION`
- `DAY_VOTE`
- `DAY_RUNOFF`
- `NIGHT`
- `WAITING_HOST_DECISION`
- `GAME_OVER`

日数は状態とは別に `day_number` で管理すること。`DAY_DISCUSSION` などに day number を持たせてよいが、実装上は `phase + day_number` の組で扱ってよい。

### 基本遷移
必ず次の順で進めること。

1. `LOBBY`
2. `SETUP`
3. `NIGHT_0`
4. `DAY_DISCUSSION` day 1
5. `DAY_VOTE` day 1
6. 必要なら `DAY_RUNOFF` day 1
7. `NIGHT` day 1
8. `DAY_DISCUSSION` day 2
9. 以後繰り返し
10. 勝利条件成立で `GAME_OVER`

未提出が残ったまま締切を迎えた場合のみ、通常遷移ではなく `WAITING_HOST_DECISION` に入ること。

### `SETUP`
- 参加者確定
- 9 席に足りない人数を LLM で埋める
- 役職をシャッフルして割り当てる
- SQLite に試合初期状態を保存する
- Discord 側チャンネル権限を初期化する

### `NIGHT_0`
以下を順番に処理すること。
1. 全員へ役職通知
2. 占い師へ初日ランダム白を通知
3. 人狼同士へ相方通知
4. 狂人には人狼情報を渡さない
5. 初夜襲撃はなし
6. 騎士の護衛もなし
7. 霊媒通知もなし
8. 必要なログを保存して `DAY_DISCUSSION` day 1 へ進む

初日ランダム白の対象は以下を満たすこと。
- 生存中
- 占い師本人ではない
- 人狼ではない

### `DAY_DISCUSSION`
- 1 日目は 300 秒
- 2 日目は 240 秒
- 3 日目以降は 180 秒固定
- bot は開始時に日数、残り人数、前夜結果、議論締切時刻を投稿する
- 死亡者の役職は公開しない。名前と死亡理由だけを公開する
- 人間は VC で会話してよいが、LLM と共存する場合は以下をメイン text に流す運用を前提にする
  - CO
  - 占い結果
  - 霊媒結果
  - 質問
  - 強い疑い
  - 要約

### LLM の昼行動
- 各生存 LLM は `DAY_DISCUSSION` 開始後に 1 回、最初の発言を行う
- その後は以下の条件で追加入力を行う
  - 自分宛ての質問やメンションが来たとき
  - 誰かが CO や判定結果を出したとき
  - 議論中盤を過ぎてもその LLM が未発言の論点があるとき
- 同一 LLM の通常発言は 1 日あたり最大 3 回までに制限する
- 投票直前に 1 回だけ短い投票意思表明をしてよい
- 連投防止のため、通常発言には 20 秒以上のクールダウンを入れる

### `DAY_VOTE`
- 生存者のみ投票
- 自分自身への投票は禁止
- 投票は DM で秘密投票
- 未投票者が締切までに残った場合、自動確定せず `WAITING_HOST_DECISION` に遷移する
- ホストが `/wolf force-skip` を実行した場合、未投票者は棄権として確定する
- 投票集計は提出済み票のみで行う
- 最多得票者が 1 人なら処刑
- 最多得票者が複数なら `DAY_RUNOFF`

### `DAY_RUNOFF`
- 最多同票者のみを候補にした決選投票を 1 回だけ行う
- 投票者はその時点の生存者全員
- 自分自身への投票は禁止
- 未投票処理は `DAY_VOTE` と同じ
- 再度同票なら、その日は処刑なしで `NIGHT` へ進む

### 処刑後
- 処刑者を死亡状態に更新
- メイン text の発言権を外す
- 天国部屋の発言権を付与する
- 勝利判定を実行する
- 続行なら `NIGHT` へ進む

### `NIGHT`
夜フェイズでは以下を行うこと。
- 人狼: 襲撃対象を 1 名選択
- 占い師: 占い対象を 1 名選択
- 騎士: 護衛対象を 1 名選択
- 霊媒師: 当日処刑されたプレイヤーの陣営結果を受け取る

### 夜の合法対象
- 人狼の襲撃対象: 生存中の非人狼、自分以外
- 占い対象: 生存中の自分以外
- 騎士の護衛対象:
  - 生存中
  - 自分以外
  - 前夜に護衛した相手以外

### 騎士
- 自分護衛不可
- 同一対象の連続護衛不可を既定にする
- 前夜の護衛先を保存し、翌夜の候補一覧から除外すること
- 初夜護衛なし

### 人狼相談
- 人狼専用テキストチャンネルは夜のみ使用可能
- 生存中の人狼のみ閲覧と発言を許可する
- 昼は書き込み不可にする
- 人狼が 2 人とも生存している場合、襲撃対象は各人が個別提出する
- 2 票が一致した場合、その対象を襲撃する
- 1 対 1 で割れた場合、締切時点では未確定のままとし、`WAITING_HOST_DECISION` に遷移する
- ホストが `/wolf force-skip` を実行した場合、割れた襲撃は不成立として処理する
- 生存人狼が 1 人だけなら、その人の選択を採用する

### 霊媒師
- 霊媒師は当日処刑されたプレイヤーが `人狼陣営` か `村人陣営` かだけを知る
- 役職名そのものは通知しない
- 処刑なしの日は `本日の霊媒結果なし` を通知する

### 夜の締切
- 未提出がある場合、自動的に結果確定して次フェイズへ進めてはいけない
- ただし霊媒師の通知のような受動情報は未提出扱いに含めない
- 未提出があれば `WAITING_HOST_DECISION` に遷移し、bot が以下を明示する
  - 現在フェイズ
  - 未提出者
  - 未確定の行動種類
  - `/wolf extend` または `/wolf force-skip` の案内

### `/wolf force-skip` 時の扱い
- `DAY_VOTE` / `DAY_RUNOFF`: 未提出者は棄権
- `NIGHT` の人狼襲撃: 未提出者は行動なし。複数狼で票が割れたままなら襲撃不成立
- `NIGHT` の占い師: 行動なし
- `NIGHT` の騎士: 行動なし

### 夜の解決順
解決順は必ず以下に固定すること。
1. 当日処刑者の霊媒結果を霊媒師へ通知
2. 占い結果を確定
3. 護衛先を確定
4. 襲撃先を確定
5. 襲撃先と護衛先が一致していれば死亡なし
6. 一致しなければ襲撃対象を死亡にする
7. 朝の公開アナウンスを投稿
8. 死亡者のチャンネル権限を更新
9. 勝利判定
10. 続行なら次の `DAY_DISCUSSION`

占いは夜開始時点の生存者に対して有効とする。夜の襲撃で朝に死亡する相手を占った場合でも、占い結果は得られる。

### 朝アナウンス
- 公開するのは以下のみ
  - 平和かどうか
  - 死亡者名
  - 次の昼フェイズ開始
- 役職名は公開しない

## 天国部屋
- 死亡者のみ閲覧・発言可能
- ゲーム中の秘密役職を勝手に bot が全公開してはいけない
- 天国部屋には以下を投稿してよい
  - 死亡通知
  - 公開情報の再掲
  - 死者向け雑談案内
- 死亡した LLM プレイヤーも天国部屋で軽く会話を続けてよい
- 生存者は天国部屋を見られないようにする

## LLM 実装要件

### 基本方針
- LLM はゲームに勝つことを目指して行動する
- ただし知識は自分の役職で見えている情報だけに限定する
- 人狼なら嘘をついてよい
- 狂人も嘘をついてよい
- 村陣営は原則として秘密情報を捏造しない
- 人格は保つが、ゲーム上不利になるほどロールプレイを優先してはいけない

### LLM 入力
少なくとも以下をプロンプトに入れること。
- 自分の名前
- 人格説明
- 自分の役職
- 生存者一覧
- 死亡者一覧
- 既知の私的情報
- 公開ログ要約
- 当日フェイズ
- 投票履歴
- 自分の前回発言

### LLM 出力
構造化出力で最低でも以下を返させること。
- `intent`: `speak`, `vote`, `night_action`, `skip`
- `public_message`
- `target_name`
- `reason_summary`
- `confidence`

`reason_summary` は内部運用用の短い説明にし、長い思考ログを保存しないこと。

### LLM の発言ルール
- 1 発言は 80 文字から 300 文字程度を目安にする
- 日本語で話す
- キャラらしさは口調と判断傾向で出す
- 原作の台詞をそのまま再現しない
- メタ発言や AI であることの自己言及は禁止

## 永続化

### SQLite に保存するもの
最低でも以下を保存すること。
- game room
- seat
- player
- persona
- role
- alive/dead 状態
- current phase
- day number
- current deadline
- submitted votes
- submitted night actions
- previous guard target
- public logs
- private logs
- host intervention pending state

### 復旧
- 起動時に進行中ゲームを読み込む
- 締切時刻を過ぎていた場合、勝手に先へ進めず `WAITING_HOST_DECISION` に移す
- 復旧後に Discord 側チャンネル権限と内部状態が一致しているか補正する
- 復旧に失敗したときはゲームを壊さず、ホストに管理者向けエラーメッセージを出す

## テスト要件
少なくとも以下を自動テストで固定すること。
- 役職配布が常に `2/1/1/1/1/3` である
- 人間不足時に LLM がちょうど不足分だけ入る
- Gnosia 人格名が重複しない
- 初日ランダム白が占い師本人や人狼を選ばない
- 昼タイマーが `300 -> 240 -> 180 -> 180 ...` で進む
- 秘密投票の最多同票時に決選へ進む
- 決選再同票で処刑なしになる
- 騎士が自分を護衛できない
- 騎士が同じ相手を連続護衛できない
- 護衛成功時に朝の死亡者が出ない
- 襲撃と護衛の解決順が固定されている
- 霊媒師が役職名ではなく陣営だけを受け取る
- 死亡者が天国部屋に入り、メイン text で話せなくなる
- 人狼専用チャットが夜だけ有効
- 未提出時に自動進行せず `WAITING_HOST_DECISION` に止まる
- 再起動復旧後に phase, actions, deadlines が壊れない

## 静的解析と品質基準
- `ruff check` が通ること
- `mypy` が通ること
- `pytest` が通ること
- 例外を握りつぶさないこと
- Discord API 呼び出し失敗時はログとユーザー向けメッセージを分けること
- 状態遷移ロジックはテスト可能な純粋関数またはそれに近い形へ寄せること

## 実装上の注意
- Discord の UI 更新と内部 state 更新を分離し、内部 state を正とすること
- 1 試合同時実行しか対応しないなら、それを明示して壊れないように実装すること
- 複数試合を許可するなら guild/channel 単位で厳密に分離すること
- タイマーは `asyncio.sleep` の生ループに依存しすぎず、復旧可能な deadline ベースで設計すること
- 「一定時間後に次へ進む」ではなく「deadline に達し、かつ未提出がないなら次へ進む」という条件で遷移すること
- 権限更新失敗時に内部状態だけ先へ進めないこと
- ログは公開ログと秘密ログを分けること
