# 人狼bot コードレビュー結果

## 概要

- 対象: `src/wolfbot`, `tests`, `CLAUDE.md`, `prompts/IMPLEMENTATION_PROMPT.md`
- 観点: 状態遷移、永続化の整合性、Discord/LLM 境界での入力受理、復旧時の安全性
- 結論: テスト・静的解析はすべて通過している一方で、実運用では破綻しうる高優先度の欠陥が 2 件、重複通知の不整合が 1 件あります

## 実施した確認

- `uv run pytest tests -q` → `127 passed in 0.54s`
- `uv run ruff check src tests` → `All checks passed!`
- `uv run mypy` → `Success: no issues found in 24 source files`
- 追加で一時 DB を使った再現スクリプトを実行し、遷移途中の例外時の整合性崩壊と、不正な提出内容の受理を確認

## 指摘事項

### High: `apply_transition()` が単一トランザクションになっておらず、途中失敗で状態が壊れる

- 影響: フェイズ更新だけが commit され、プレイヤー更新やログ保存が失敗すると、DB 上のフェイズと実データが食い違います。以後の復旧・進行・権限同期が壊れます。
- 根拠コード:
  - `src/wolfbot/persistence/sqlite_repo.py:573`
  - `src/wolfbot/persistence/sqlite_repo.py:588`
  - `src/wolfbot/persistence/sqlite_repo.py:593`
- 詳細: `games.phase/day_number/deadline_epoch` の更新だけを `_tx()` 内で commit し、その後の `force_skip` 解除、`player_updates`、ログ、`pending_decisions`、`previous_guard` 更新はトランザクション外で続けています。
- 再現:
  - `_apply_player_update()` を例外化すると、`apply_transition()` 実行後に `phase='SETUP'` だけが残り、`seat1_role` は `None` のままでした。
  - これは「フェイズだけ進んだが配役や付随データが存在しない」破損状態です。
- 改善案:
  - `apply_transition()` 全体を 1 つの `_tx()` に戻し、phase 更新から付随テーブル更新までを原子的に扱うべきです。
  - 少なくとも `games` 更新成功後に起こる例外では全ロールバックされる必要があります。

### High: 古い DM / 不正 target の投票・夜行動がそのまま受理され、死者を再度処刑・襲撃できる

- 影響: フェイズをまたいだ古い UI 操作や、候補外・死亡済み座席への提出を保存できてしまいます。その結果、すでに死亡したプレイヤーが再度 kill 扱いになり、`death_day` が上書きされます。
- 根拠コード:
  - `src/wolfbot/services/game_service.py:356`
  - `src/wolfbot/services/game_service.py:379`
  - `src/wolfbot/domain/rules.py:97`
  - `src/wolfbot/domain/state_machine.py:388`
  - `src/wolfbot/domain/state_machine.py:566`
- 詳細:
  - `submit_vote()` / `submit_night_action()` は current phase、day、actor の生死、target の合法性を検証せず、その時点の `game.day_number` で保存します。
  - `compute_vote_result()` は「投票先が生存者か」を見ていないため、死亡済み seat への票も通常投票では有効票になります。
  - `_apply_execution()` と `plan_night_resolve()` も、対象がすでに死んでいるかを防御していません。
- 再現:
  - すでに死亡済みの seat 7 に票を集めると、ゲームは `GAME_OVER` に進み、seat 7 の `death_day` が 2 に上書きされました。
  - 夜行動でも、死亡済みの seat 7 を襲撃対象として保存すると、朝遷移後に `death_day` が 2 に更新されました。
- 改善案:
  - `submit_vote()` / `submit_night_action()` で phase・round・day・actor・target を検証し、現在の合法候補以外は reject するべきです。
  - ルール層でも target が現在の生存候補に含まれるか再確認し、保存済み不正データがあっても無効化する防御が必要です。

### Medium: 夜明けメッセージが二重送信される

- 影響: 朝の通知が同じ遷移で 2 回送られ、ユーザーから見ると冗長で、ログの意味づけも曖昧になります。
- 根拠コード:
  - `src/wolfbot/domain/state_machine.py:590`
  - `src/wolfbot/domain/state_machine.py:609`
  - `src/wolfbot/services/game_service.py:193`
  - `src/wolfbot/services/game_service.py:201`
- 詳細:
  - `plan_night_resolve()` は朝文面を `public_logs(kind="MORNING")` と `morning_text` の両方に積んでいます。
  - `GameService._advance_once()` は `public_logs` を `post_public()` し、その後 `morning_text` を `post_morning()` しています。
- 再現:
  - フェイク Discord では同一遷移で `post_public:MORNING` と `post_morning` の両方が記録されました。
- 改善案:
  - 朝通知は `public_logs` か `morning_text` のどちらか一方に統一するべきです。
  - UI 上の装飾を変えたいだけなら、保存表現と送信表現の責務分離を明確にしてください。

## 補足

- 現在のテストはかなり厚く、通常フローの回帰には効いています。
- ただし今回の 3 件は、いずれも「異常系または古い UI/不正入力に対する防御」が未カバーなため、`pytest` 全通だけでは検出できていません。
- 次に手を入れるなら、優先順位は `apply_transition()` の原子性修正、提出時バリデーションの追加、その後に朝通知の整理です。
